System Architecture

How the Smart Router Works

The Smart Router (protocol/rpcsmartrouter/) is an enterprise RPC aggregation layer that sits between clients (custodians, exchanges, wallets) and a pool of RPC node providers. It abstracts multi-provider routing behind a single endpoint, selecting the best provider per request using a weighted composite QoS score, and retrying failed requests transparently.

Risk Analysis

Five Critical Technical Risks

Ranked by enterprise blast radius. The first three will lose contracts or cause incidents. The last two erode trust over time.

Critical Risk 1: No Graceful Shutdown or Health Signaling

The signal handler catches only SIGINT, ignoring SIGTERM entirely.

signalChan := make(chan os.Signal, 1)
signal.Notify(signalChan, os.Interrupt)  // SIGINT only
<-signalChan
return nil

rpcsmartrouter.go:300-302

1a. SIGTERM is ignored. Kubernetes sends SIGTERM for pod eviction and rolling deploys. The process doesn't catch it. K8s waits 30s, then SIGKILL. Every rolling deploy is a hard kill, wiping dedup cache, dropping in-flight requests, resetting QoS state.
1b. No connection drain. No mechanism to stop accepting new connections while finishing in-flight relays. A custodian's eth_sendRawTransaction killed mid-relay can cause double-submission on retry.
1c. No health/liveness endpoint. No /health, no gRPC health service, no readiness probe. A router stuck in a lock storm still accepts TCP connections but never responds. Load balancer keeps sending traffic to a zombie.
Every deploy is a potential double-execution event
Low-medium, about a week
Critical Risk 2: Retry Dedup Cache Lost on Every Restart

In-memory ristretto cache with 6-hour TTL. Combined with Risk 1 (no graceful shutdown), every deploy wipes this cache.

const (
    CacheMaxCost     = 10 * 1024
    CacheNumCounters = 20000
    RetryEntryTTL    = 6 * time.Hour
)

relay_retries_manager.go:11-14

The RelayRetriesManagerInf interface already has the right abstraction: three methods, no complex semantics. A Redis backend is straightforward.

relay_retries_manager.go:17-21

Highest-severity hidden risk: double-execution for custodians
Low, about 2 days
High Risk 3: Lock Contention Cascade Under Traffic Spikes

Every relay traverses two nested locks:

1. UsedProviders.lock, polled via TryLockSelection up to 500 × 5ms = 2.5s max wait used_providers.go:12, 230-250

2. ConsumerSessionManager.lock (RWMutex), guards 8 fields including pairing map consumer_session_manager.go:41

for counter := 0; counter < MaximumNumberOfSelectionLockAttempts; counter++ {
    // ...
    time.Sleep(5 * time.Millisecond)
}

Under a traffic spike (1,000+ concurrent goroutines), a single write event starves all readers. No backpressure mechanism; the router accepts unlimited inbound connections. Self-reinforcing: errors trigger retries that create more goroutines.

P99 → ∞ during spikes. SLA breach.
Load test first, validate before committing to lock decomposition
High Risk 4: Circuit Breaker Is Primitive and Unobservable 
if srsm.consecutivePairingErrors >= 2 {
    utils.LavaFormatWarning("Circuit breaker triggered: ...")
    go validateReturnCondition(err)
    continue
}

smartrouter_relay_state_machine.go:378

Problems:

Inline literal threshold (2), not configurable. Operators cannot tune it.
Counter resets on error-type change line 391. A flapping provider prevents the circuit from ever opening.
No half-open state. Circuit snaps closed instantly. No probe requests, no backoff.
No Prometheus metric. State transitions emit only LavaFormatWarning, invisible to production alerting.
Thousands of futile retries during provider outages, operator unaware
Medium, use sony/gobreaker

Failure Analysis

Production Stress Scenarios

Not restatements of risks. Specific scenarios where multiple risks compound.

Scenario A: The Rolling Deploy
Risks 1a + 1c + 2 compound on every deploy
1
Ops runs kubectl rollout restart during peak hours. Kubernetes sends SIGTERM.
2
Router ignores SIGTERM (Risk 1a). After 30s, SIGKILL fires. In-memory QoS data and dedup cache are destroyed.
3
Replacement pod starts blank, so all providers look identical. First ~5,000 requests route near-randomly while T-Digest rebuilds.
!
Double-execution risk for every subscription in flight at the moment of SIGKILL. Not an edge case. The normal operational path.
Scenario B: The Provider Cascade
One provider failure spirals into multi-provider degradation
1
1 of 4 providers loses connectivity. Detection takes 5 × 1500ms = 7.5 seconds. common.go:37-38
2
25% of goroutines pile up waiting for the dead provider. BACKOFF_TIME_ON_FAILURE = 3s adds delay. Total degradation: 10-12 seconds. common.go:45
3
Remaining 3 providers absorb 33% more traffic. If one was at 80% capacity, it starts timing out.
4
Circuit breaker's flapping-reset bug (Risk 4) prevents it from opening. Two providers now degraded.
!
System spirals. At 1,000 TPS: 10,000-12,000 failed or delayed requests during the detection window.
Scenario C: The Lock Storm
Traffic spike + pairing update = self-reinforcing cascade
1
Sudden traffic spike (exchange listing, token launch) coincides with a pairing update.
2
Write lock in UpdateAllProviders() starves all readers. Goroutines accumulate in TryLockSelection's 5ms polling loop. used_providers.go:230
3
No backpressure, so new connections keep arriving. Provider timeouts generate errors → trigger retries → spawn more goroutines.
!
Go scheduler degrades under goroutine pressure. P99 → ∞. Hard to reproduce in staging. Requires load testing at 2× production TPS with simultaneous provider kill.

Hardening Plan

Prioritized 90-Day Roadmap

Sequenced by blast radius, not elegance. Fix what pages you at 3am first.

Month 1

Stop the Bleeding

▼

Week 1-2: Graceful shutdown + health

  • Add syscall.SIGTERM to signal handler rpcsmartrouter.go:301
  • Connection drain: stop accepting, finish in-flight with configurable timeout (15s default)
  • Expose /health endpoint: 200 when ready, 503 once drain begins
  • K8s liveness/readiness probe config

Week 2-3: Prometheus metrics

  • Circuit breaker state transitions smartrouter_relay_state_machine.go:378
  • TryLockSelection wait-time histogram used_providers.go:230
  • CU mismatch events single_provider_session.go:98
  • Provider block/unblock events with reason

Week 3-4: Load test & profile

  • 15-min test at 2× production TPS
  • Kill one provider at minute 5, restore at minute 10
  • Profile pprof goroutine dumps at peak
  • Result determines if lock decomposition enters Month 2

Strategic Discipline

What We Would Not Focus On

All reasonable engineering projects. None reduces the probability of a 3am page or an SLA breach in the next 90 days.

</>
ChainLib / gRPC Proxy Refactoring
Deliberately excluded from linting in .golangci.yml. Known-messy, but not on the enterprise reliability hot path. High regression risk, low blast-radius reduction. Q2.
✓
CrossValidation Mode Hardening
Complex failure modes (split consensus, quorum never reached). But Stateless mode carries 95%+ of enterprise traffic. Hardening CrossValidation before Stateless is reliable means optimizing the wrong path.
Σ
WRS Algorithm Improvements
The adaptive P10-P90 normalization via T-Digest is already solid and recently shipped weighted_selector.go. The problem is missing observability and wrong defaults, not algorithm quality. Tuning an algorithm you can't observe is guesswork.
⇄
Horizontal Smart Router Clustering
No shared session state between instances. A load balancer in front of single-instance routers provides horizontal scale today. Design clustering after single-instance reliability is proven. Adding distributed coordination before you need it creates more problems than it solves.
⚙
Relay State Machine Rewrite
SmartRouterRelayStateMachine is complex but well-covered by tests. The right move is targeted: add metrics, fix the circuit breaker, make constants configurable. Restructuring the state machine while also changing its behavior compounds regression risk.
